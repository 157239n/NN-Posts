Today, we will recreate probably the most useful tool that completely defines PyTorch: Autograd. The objective is to recreate every functionality of autograd only with basic tensor manipulation tools, without any PyTorch's other magical functions. The hope is to give you a sense of independence from the platform itself and a feeling that you can rebuild everything from scratch.

We have already seen how to build other basic components from scratch in a previous post, but we still use Autograd as a blackbox. The goal is to eliminate that last blackbox. So without further ado, let's import some stuff:

What is Autograd?¶

Autograd is a mechanism to automatically calculate gradients as you do the math. For example, we have 4 variables, $a, b, c, d$, where $b = f(a); c=g(b); d=h(c)$. You only need to calculate $b, c, d$, and Autograd will automatically calculate for you $\frac{\partial d}{\partial a}, \frac{\partial d}{\partial b}, \frac{\partial d}{\partial c}$, which you can use directly in your optimizer.

Let's go through an example. Let's define a tensor drawn from the uniform distribution $U(0, 1)$:

In [63]:
a = torch.rand(3).requires_grad_(True); a
Out[63]:
tensor([0.3675, 0.8485, 0.8939], requires_grad=True)

Let's now calculate other values:

In [64]:
b = 2 * a
c = torch.exp(b)
d = 3 + c
e = d.sum()
b, c, d, e
Out[64]:
(tensor([0.7350, 1.6969, 1.7878], grad_fn=<MulBackward0>),
 tensor([2.0856, 5.4572, 5.9764], grad_fn=<ExpBackward>),
 tensor([5.0856, 8.4572, 8.9764], grad_fn=<AddBackward0>),
 tensor(22.5192, grad_fn=<SumBackward0>))

Now let's get the gradients:

In [65]:
e.backward(); a.grad
Out[65]:
tensor([ 4.1711, 10.9144, 11.9529])

So here,

  • $b_i=2\cdot a_i$
  • $c_i=e^{b_i}$
  • $d_i=3+c_i$
  • $e=\sum_i^3d_i$

Now, let's calculate the gradients by hand:

  • $\frac{\partial e}{\partial d_i} = 1$
  • $\frac{\partial e}{\partial c_i}=\frac{\partial e}{\partial d_i}\cdot\frac{\partial d_i}{\partial c_i}=1\cdot1=1$
  • $\frac{\partial e}{\partial b_i}=\frac{\partial e}{\partial d_i}\cdot\frac{\partial d_i}{\partial c_i}\cdot\frac{\partial c_i}{\partial b_i}=1\cdot1\cdot e^{b_i}=e^{b_i}$
  • $\frac{\partial e}{\partial b_i}=\frac{\partial e}{\partial d_i}\cdot\frac{\partial d_i}{\partial c_i}\cdot\frac{\partial c_i}{\partial b_i}\cdot\frac{\partial b_i}{\partial a_i}=1\cdot1\cdot e^{b_i}\cdot2=2e^{b_i}$

Let's check whether this is the same as a.grad:

In [66]:
2*torch.exp(b)
Out[66]:
tensor([ 4.1711, 10.9144, 11.9529], grad_fn=<MulBackward0>)

Yep, it has automatically calculate every gradients for us. This is pretty awesome right? When I was starting out with neural networks, I got really, really turned off whenever I have to manually calculate the partial derivatives for every function out there. I mean, for simple activations like ReLU, the derivative is easy. 0 or 1, and that's it. But for stuff like sigmoid, it seems like I have to derive the derivative equation by hand, then code that in somehow, and overall it's quite a mess. And people like other stuff too, like tanh, at which point I just gave up. The first time I heard about Autograd I couldn't believe it too. It streamlines everything so, so much, and now literally the sky is the limit.

So now, the gradient of interest is $\frac{\partial e}{\partial a_i}=\frac{\partial e}{\partial d_i}\cdot\frac{\partial d_i}{\partial c_i}\cdot\frac{\partial c_i}{\partial b_i}\cdot\frac{\partial b_i}{\partial a_i}$. Individual terms:

  • $\frac{\partial e}{\partial d_i}=1$
  • $\frac{\partial d_i}{\partial c_i}=cos(c_i)$
  • $\frac{\partial c_i}{\partial b_i}=e^{b_i}$
  • $\frac{\partial b_i}{\partial a_i}=2$

So, $\frac{\partial e}{\partial a_i}=2\cdot cos(c_i)\cdot e^{b_i}$. Let's test it:

Yep, exactly the same as a.grad. Let's now build Autograd right away.

Autograd for simple functions¶

In [828]:
def IdentityBackward(x1, x2, nextGrad): return (x1, None)
def ExpBackward(x1, x2, nextGrad): return (nextGrad * torch.exp(x1.data), None)
def SinBackward(x1, x2, nextGrad): return (nextGrad * torch.cos(x1.data), None)
def CosBackward(x1, x2, nextGrad): return (nextGrad * (-torch.sin(x1.data)), None)
def LogBackward(x1, x2, nextGrad): return (nextGrad * 1/x1.data, None)
def MulBackward(x1, x2, nextGrad): return (nextGrad * x2.data, nextGrad * x1.data)
def AddBackward(x1, x2, nextGrad):
    return (nextGrad * torch.ones(*x1.shape), nextGrad * torch.ones(*x2.shape))
def SumBackward(x1, x2, nextGrad): return (nextGrad * torch.ones(*x1.shape), None)
In [829]:
class Tensor:
    def __init__(self, data, grad_fn=None):
        self.data = data; self.shape = data.shape
        self.requires_grad = False; self.grad_fn = grad_fn; self.grad = None
        self.x1, self.x2 = None, None # placeholders for dependent variables
    def requires_grad_(self, requires_grad):
        self.grad_fn = IdentityBackward if requires_grad else None
        self.requires_grad = requires_grad; return self
    def __repr__(self):
        return str(self.data)
    def sum(self, dim=None):
        if dim is not None:
            if self.grad_fn:
                answer = Tensor(self.data.sum(dim=dim), grad_fn=SumBackward)
                answer.x1 = self; return answer
            else: return Tensor(self.data.sum(dim=dim))
        if self.grad_fn:
            answer = Tensor(self.data.sum(), grad_fn=SumBackward)
            answer.x1 = self; return answer
        else: return Tensor(self.data.sum())
    def __add__(self, tensor):
        if type(tensor) in [int, float]: tensor = Tensor(torch.ones(*self.shape)*tensor)
        #if self.shape != tensor.shape: raise Exception("Incompatible shapes")
        if tensor.grad_fn or self.grad_fn:
            answer = Tensor(self.data + tensor.data, grad_fn=AddBackward)
            answer.x1 = self; answer.x2 = tensor; return answer
        else: return Tensor(self.data + tensor.data)
    def __radd__(self, tensor): return self.__add__(tensor)
    def __mul__(self, tensor):
        if type(tensor) in [int, float]: tensor = Tensor(torch.ones(*self.shape)*tensor)
        if self.shape != tensor.shape:
            raise Exception(f"Incompatible shapes: {self.shape}, {tensor.shape}")
        if tensor.grad_fn or self.grad_fn:
            answer = Tensor(self.data * tensor.data, grad_fn=MulBackward)
            answer.x1 = self; answer.x2 = tensor; return answer
        else: return Tensor(self.data * tensor.data)
    def __rmul__(self, tensor): return self.__mul__(tensor)
    def calculateGradientOfDependents(self):
        if self.grad_fn:
            gradX1, gradX2 = self.grad_fn(self.x1, self.x2, self.grad)
            # this is the chain rule part
            if type(self.grad) != type(None):
                if type(gradX1) != type(None): self.x1.grad = gradX1
                if type(gradX2) != type(None): self.x2.grad = gradX2
                # if doesn't retain gradientstrack track , then delete to free resources
                if not self.requires_grad: del self.grad; self.grad = None
            if type(self.x1) != type(None): self.x1.calculateGradientOfDependents()
            if type(self.x2) != type(None): self.x2.calculateGradientOfDependents()
    def backward(self):
        if len(self.shape) != 0:
            raise Exception(".backward() can only be called from rank 0 tensors")
        self.grad = 1; self.calculateGradientOfDependents()
        gc.collect()
class Utils:
    def __init__(self): pass
    def rand(self, *shape): return Tensor(torch.rand(shape))
    def zeros(self, *shape): return Tensor(torch.zeros(shape))
    def ones(self, *shape): return Tensor(torch.ones(shape))
    def exp(self, tensor: Tensor):
        if tensor.grad_fn:
            answer = Tensor(torch.exp(tensor.data), grad_fn=ExpBackward)
            answer.x1 = tensor; return answer
        else: return Tensor(torch.exp(tensor.data))
    def sin(self, tensor: Tensor):
        if tensor.grad_fn:
            answer = Tensor(torch.sin(tensor.data), grad_fn=SinBackward)
            answer.x1 = tensor; return answer
        else: return Tensor(torch.sin(tensor.data))
    def cos(self, tensor: Tensor):
        if tensor.grad_fn:
            answer = Tensor(torch.cos(tensor.data), grad_fn=CosBackward)
            answer.x1 = tensor; return answer
        else: return Tensor(torch.cos(tensor.data))
    def log(self, tensor: Tensor):
        if tensor.grad_fn:
            answer = Tensor(torch.log(tensor.data), grad_fn=LogBackward)
            answer.x1 = tensor; return answer
        else: return Tensor(torch.log(tensor.data))
utils = Utils()

And the result is exactly that! Isn't it wonderful? Now let's understand what's going on.

First thing first, we're defining our own Tensor class. Just like torch.Tensor, Tensor will have 2 torch.Tensor objects. 1 is data for the main tensor and the other is grad, for the gradient of the main tensor. grad is set to None in the first place though, because there are no gradients initially. Only when we call .backward() will the gradient appear, and we can do things with it. Tensor will also have a requires_grad variable, to indicate that we should track and retain the gradient of this tensor. requires_grad can be set using the .requires_grad_() function, just like how torch.Tensor did it. All derived tensors from a tensor that retains the gradient tracks, but doesn't have to retain the gradients. Meaning, if $a$ has requires_grad set to true, and $b = sin(a), c = log(b)$, then $b$ and $c$ will track their own gradients, but don't have to retain the gradients.

The Utils class with an accompanying utils singleton object gives us a bunch of functions, to make it parallel to PyTorch's way of doing it. For example, utils.exp() looks and feels like torch.exp(). Utils.rand(), Utils.zeros() and Utils.ones() are pretty straightforward. It takes in a number of dimensions, and then creates a Tensor with data from the respective PyTorch's functions.

Now I want you to focus on Utils.exp(), Utils.sin(), Utils.cos(), and Utils.log(), but only the last line of each function, where the gradients are not tracked. Tensor.grad_fn is not None only if we have decided to track the gradient (but not necessarily retain it), and so we can use that to check whether we need to deal with all the gradient stuff for a particular operation or not. All of them look simple enough. We're just creating a new Tensor with the transformed data and returning it back. Pretty straightforward.

Let's jump into fire and look at this dynamic computational graph:

They don't have to always have a value though. If the operation has 2 variables, like $y+z$, then Tensor.x1 will point to $y$, and Tensor.x2 will point to $z$. If the operation has 1 variable only, like $e^x$, then Tensor.x1 will point to $x$, and Tensor.x2 will point to None. If the operation has 0 variables (tensor initialization), like with tensor a above, then both Tensor.x1 and Tensor.x2 point to None.

Let's now zoom in and just consider tensors a and b. a's local gradient ($\frac{d}{da}b$) is just b.grad_fn(a, None), but the global gradient ($\frac{d}{da}e$) will be a.grad = b.grad_fn(a, None) * b.grad ($\frac{d}{da}e=\frac{d}{da}b\cdot\frac{d}{db}e$). We can combine these behavior so that to get the global gradient we can call a.grad = b.grad_fn(a, None, b.grad) This logic is implemented in the Tensor.calculateGradientOfDependents() function. Then we can write Tensor.backward() as a wrapper function for that internal function, to deal with logistical stuff.

We have devised a way to calculate c.grad from d.grad, a.grad from b.grad, etc. So, e.grad is first assigned the number 1, then d.grad can be calculated, then c.grad and so on, until Tensor.x1 and Tensor.x2 are both None, in which case the original tensor whose gradient we're keeping track of already has its gradient calculated.

Computers nowadays are extremely fast, but we're trying to simulate brains here, and currently, biological brains outperform artificial ones by a huge margin, as explored in a previous post, so we still have to be lean and mean about CPU cycles and RAM. So, after we have used e.grad to calculate d.grad, if tensor d doesn't have to retain the gradient, we can just delete e.grad right away, and we can still calculate c.grad by using d.grad. This extra bit is implemented right within Tensor.calculateGradientOfDependents(), and the wrapper function Tensor.backward() calls the garbage collector to free deleted tensors.

Reverse engineering complex, multidimensional functions¶

In the previous section, we're just dealing with simple functions that don't change the shape of the tensor. $e^x, log(x), sin(x), cos(x), x + 3$ all have the same shape as $x$. The summation actually changes the shape a little bit, from shape $(3,)$ to a single number (0 dimension), but that's sort of easy. There's another type of function we haven't dealt with yet, and that is matrix multiplication, especially the high dimensional ones so let's dive right in:

Turns out, b.sum(dim=1) is actually the rows in a.grad, and a.sum(dim=0) is actually the columns in b.grad. Very interesting. Let's step over this manually, to see what's going on:

$\left[\begin{array}{lllll} a_{00}&a_{01}&a_{02}&a_{03}&a_{04}\\ a_{10}&a_{11}&a_{12}&a_{13}&a_{14}\\ a_{20}&a_{21}&a_{22}&a_{23}&a_{24} \end{array}\right]\cdot\left[\begin{array}{llll} b_{00}&b_{01}&b_{02}&b_{03}\\ b_{10}&b_{11}&b_{12}&b_{13}\\ b_{20}&b_{21}&b_{22}&b_{23}\\ b_{30}&b_{31}&b_{32}&b_{33}\\ b_{40}&b_{41}&b_{42}&b_{43}\\ \end{array}\right]=\left[\begin{array}{llll} c_{00}&c_{01}&c_{02}&c_{03}\\ c_{10}&c_{11}&c_{12}&c_{13}\\ c_{20}&c_{21}&c_{22}&c_{23}\\ \end{array}\right]$

Let's write out an explicit sum, so you will be comfortable about it:

$c_{12}=\sum_i^4a_{1i}b_{i2}$. Looks reasonable?

Let's calculate some gradients:

$\frac{d}{da_{10}}c_{12}=\frac{d}{da_{10}}\sum_i^4a_{1i}b_{i2}=b_{02}$

But we've just seen that a.grad's rows are composed from b.sum(dim=1), and we didn't see any sum, so what's up with that? In code, we were actually calculating $\frac{d}{da_{10}}d$ instead of $\frac{d}{da_{10}}c_{12}$, so let's redo it:

$\frac{d}{da_{10}}d=\frac{d}{da_{10}}\sum_{ij}^{2, 3}c_{ij}=\frac{d}{da_{10}}\sum_{i}^3c_{1i}=\sum_i^3b_{0i}$, as $c_{0j}, c_{2j}$ are independent of $a_{10}\forall j\in[0, 3]$

So there's the sum we're looking for. Also note something interesting: $\frac{d}{da_{00}}d=\frac{d}{da_{10}}d=\frac{d}{da_{20}}d=\sum_i^3b_{0i}$. This means that a.grad should have the same rows, and that's exactly what we have observed. Also note the sum. We're summing over the second dimension, hence b.sum(dim=1).

I think the math is easy to follow, but hard to keep on top of your head, so may be just remember that the gradients have either the same row or column, and that the repeating element is the sum of the other tensor over some dimension. That's enough to follow along the post.

Likewise, we have $\frac{d}{db_{00}}d=\frac{d}{db_{01}}d=\frac{d}{db_{02}}d=\frac{d}{db_{03}}d=\sum_i^3a_{i0}$.

Autograd for complex, multidimensional functions¶

Before we get to the implementation, there is another fact we should be considering. It seems like we just have to create a matmul() function and sum over specific rows and columns, as we see above. However, we are missing a deeper truth, and that is, we were analyzing the gradient of a 2-step process, with step 1 being the matrix multiply, and step 2 being the summation over every elements. If we break it down to this level, then we can still preserve the locality of the matrix multiply gradient tensor and the summation gradient tensor, and this should comply better with our way of doing the chain rule earlier:

gradA, gradB = self.grad_fn(self.x1, self.x2)
# this is the chain rule part
if type(self.grad) != type(None):
    if type(gradA) != type(None): self.x1.grad = self.grad * gradA
    if type(gradB) != type(None): self.x2.grad = self.grad * gradB
    # if doesn't track gradient, then delete to free resources
    if not self.requires_grad: del self.grad; self.grad = None
if type(self.x1) != type(None): self.x1.calculateGradientOfDependents()
if type(self.x2) != type(None): self.x2.calculateGradientOfDependents()

So, the general rule of thumb is to first broadcast a, b, c so that they have the same number of dimensions, and make sure that all dimensions except the last 2 is exactly the same. Then the expanded gradient is just gonna be a.grad = c.grad @ b.T and b.grad = a.T @ c.grad, with the transpose operation switches around the last 2 dimensions only. After all of that, we gradually reduce a.grad and b.grad using .sum(dim=0) so that their dimensions match with a and b. The full implementation of this is pretty short actually.

In [834]:
def MmBackward(x1, x2, nextGrad):
    if nextGrad is None: return (None, None)
    x1Grad = nextGrad @ x2.data.transpose(-2, -1)
    x2Grad = x1.data.transpose(-2, -1) @ nextGrad
    while len(x1Grad.shape) > len(x1.shape): x1Grad = x1Grad.sum(dim=0)
    while len(x2Grad.shape) > len(x2.shape): x2Grad = x2Grad.sum(dim=0)
    return (x1Grad, x2Grad)
def matmul(self, tensor):
    if tensor.grad_fn or self.grad_fn:
        answer = Tensor(self.data @ tensor.data, grad_fn=MmBackward)
        answer.x1 = self; answer.x2 = tensor; return answer
    else: return Tensor(self.data @ tensor.data)
Tensor.matmul = matmul

Afterword¶

Autograd is THE defining feature of PyTorch, and is arguably the feature that made it so insanely popular. Recent estimates (2019) shows that the only remaining ML framework giants are TensorFlow and PyTorch, with PyTorch being the overwhelming majority in research and TensorFlow being the majority in industry due to inertia.

In frameworks before PyTorch (like TensorFlow, Keras, Theano), they all have this notion of explicitly building a model, compile them statically to low level code, with embedded metrics and loss functions. Once you plop down a model, it's like a statue that can't easily be molded or changed. In contrast, PyTorch allows us to hack everything, and recreate every functionality to its core using just Python and the API PyTorch has given us, as we have seen throughout the unboxing the blackbox series. It looks, feels, and behaves in a way we would expect from just writing mathematical equations down and really, really lower the barrier to entry.

It's still quite crazy how this simple idea that we had just gone through in this post has really enabled such advancements in the field, and I want to close with a really insightful quote:

It is perhaps under appreciated how much machine learning frameworks shape ML research. They dont just enable machine learning research, they enable and restrict the ideas that researchers are able to easily explore. How many nascent ideas are crushed simply because there is no easy way to express them in a framework?
